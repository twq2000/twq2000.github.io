Spring

为什么要用Spring？

  • 控制反转

    通过依赖注入的方式，降低组件之间的耦合程度。由IoC容器来负责创建、管理对象的生命周期和配置。

    IoC的注入方式有两种：构造器注入、setter方法注入

  • AOP技术

    Spring支持面向切面的编程方式，将一些与业务逻辑本身无关，且被多个业务所共用的操作（事务处理、打印日志、权限控制……）抽取出来。

    这样做的好处是可以减少重复代码，并且有利于未来的可拓展性和可维护性。

    它的原理是在编译时/运行时，通过动态代理的方式实现程序功能的统一维护的一种技术。例如 AspectJ 就是在编译时进行动态代理的，SpringAOP 是在运行时进行动态代理的。

  • 支持事务管理

    Spring实现事务有两种方式：声明式和编码式。声明式事务又有两种方式：xml配置方式、@Transaction 注解方式。

bean的作用域有哪些？

  • singleton：每一个bean只创建一个实例。这是默认的作用域。

    • 单例模式是线程安全的吗？

      Spring对于单例模式的实现，使用的是 volatile+DCL 的懒汉式。这种方式本身是可以保证读操作的线程安全的。

      也就是说：如果每个线程只有读操作而没有写操作，那么其实是可以保证线程安全的；否则，如果有多个线程同时进行写操作，那么就存在线程安全问题了。

  • prototype：每次对bean的调用都会生成一个新的实例。

  • request：在一个 http request 中，一个bean对应一个实例。

  • session：在一个 http session 中，一个bean对应一个实例。

  • global session：在一个全局的 http session 中，一个bean对应一个实例。


AOP的动态代理策略是怎样的？

如果目标对象实现了接口，那么会默认采用JDK动态代理（也可以强制指定为CGLib动态代理）；否则，会强制采用CGLib动态代理。

JDK动态代理是面向接口的，CGLib动态代理是基于底层字节码来生成一个被代理对象的子类。换句话说，如果被代理类是 final 的，那么CGLib代理就会失败。


常用注解及其含义

注解作用
@Value将常量、配置文件中的变量值，注入到一个变量中
@RequestMapping用于映射web请求，搭配@Controller一起使用
@RequestParam接收请求路径中的参数，如 /user?id=123
@PathVariable接收路径参数，如 /user/123
@RequestBody接收json等格式数据，并注入给指定形参对象
@ResponseBody将返回对象以某种形式的字符串返回给前端（一般是JSON、XML）
@Qualifierby name匹配模式，搭配@Autowired一起使用，从而消除注入歧义
@Component标注在类上，表示当前类是一个组件，要交由Spring来进行管理
@Configuration标注当前类是一个Java Config类，Spring可以将它视作定义bean的配置类
@Bean标注在具有返回值的方法上，将其返回值对象作为一个bean注册入IoC容器中

SpringBoot

SpringBoot是什么？

它基于Spring框架，可以帮助使用者快速构建一个准生产级别的项目。

  1. 由于Spring框架中需要进行大量的配置，且很多都是固定重复的，因此，SpringBoot预先对这些部分已经做好了习惯性的配置，使用者可以不用或者只需进行很少的手动配置。
  2. 引入了 SpringBoot Starters 来解决依赖之间可能因版本冲突导致的问题。

@SpringBootApplication 注解的含义

@SpringBootApplication 包含了：

  • @ComponentScan

  • @SpringBootConfiguration（本身就是一种 @Configuration）

  • @EnableAutoConfiguration

    其中，最关键的是 @Import(EnableAutoConfigurationImportSelector.class)。借助它，SpringBoot可以将所有符合要求的 JavaConfig 配置类都加载到IoC容器中。


SpringBoot加载配置文件的优先级顺序？

由高到低：

  1. file: ./config/application.properties
  2. file: ./application.properties
  3. classpath: resource/config/application.properties
  4. classpath: resource/application.properties

其中，file 代表项目文件的主目录；classpath 代表项目的类路径。


SpringWeb

Restful风格是什么？

Restful就是一个资源定位及资源操作的风格。它不是标准也不是协议，只是一种风格。

  1. URI只是作为资源的代表形式

  2. 客户端使用 GET、POST、PUT、DELETE 这四种操作方式对服务端资源进行操作：GET用来获取资源，POST用来新建资源（也可以用于更新资源），PUT用来更新资源，DELETE用来删除资源

    在服务端，分别对应的注解标识是：``@GetMapping,@PostMapping,@PutMapping,@DeleteMapping`。

    以 @GetMapping 为例，它其实就是 @RequestMapping(method = RequestMethod.GET) 的缩写。其他同理。


Restful风格的好处是，它允许请求路径相同，但请求参数不同或请求方式不同的情况。如果是传统方式，如果有两个相同的映射路径就会直接报错。

在Restful风格下，根据不同的参数和请求方式，会执行不同的方法，最终产生的结果也不同，因此就不存在同路径时的冲突问题。

以下是一个示例：

如果访问 “/add?a=1&b=2”，那么在页面上就会显示 “add1:3” 字符串；

如果访问 “/add/3/4”，那么在页面上就会显示 “add2:7” 字符串；

如果访问 “/add?a=1&b=2”，且提交方式是POST请求，那么在页面上就会显示 “add3:3” 字符串。


@RestController

根据官方文档：

@RestController is a stereotype annotation that combines @ResponseBody and @Controller.

意思是：``@RestController=@ResponseBody＋@Controller`。

也就是说：使用 @RestController，意味着这个类中的所有方法上面都加了 @ResponseBody。

至于 @ResponseBody 的作用，就是将返回对象以某种形式的字符串返回给前端（一般是JSON、XML）。

总之，用 @Controller，返回的是页面；用 @RestController，返回的是JSON、XML或其他文本格式的字符串。
